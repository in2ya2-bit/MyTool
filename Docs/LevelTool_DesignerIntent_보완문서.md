## 기획의도 — 3단계 구조

| 단계 | 내용 | 상태 |
|------|------|------|
| **1단계** | 실제 지형을 토대로 한 지형·지물 생성 | ✅ 구현 완료 (기존 LevelTool) |
| **2단계** | 기획의도에 따라 지형·지물 일괄 수정 (슬라이더 + 프리셋) | 신규 개발 |
| **3단계** | 배틀로얄·익스트랙션 룰에 어울리는 옵션 (체크 + 제안) | 신규 개발 |

```
[1단계: 실사 생성] → [2단계: 의도 기반 수정] ⇄ [3단계: 룰셋 옵션]
     완료             슬라이더/프리셋          체크리스트+제안
```

### 2단계 ⇄ 3단계 자유 전환 규칙

2단계(의도 수정)와 3단계(룰셋 옵션)는 **탭 전환으로 자유롭게 오갈 수 있다**. 순서 강제 없음.

| 전환 | 시스템 동작 |
|------|-----------|
| **2단계 → 3단계** (Tab 전환) | 체크리스트 **자동 재실행** (< 3초). 2단계 수정이 반영된 유효 맵 상태 기준 분석 |
| **3단계 → 2단계** (Tab 전환) | 추가 동작 없음. 3단계 제안으로 생성된 레이어는 그대로 유지 |
| **3단계 제안 수락 후 2단계 슬라이더 조정** | 제안 레이어는 독립 유지. 슬라이더는 유효 맵 상태(제안 포함) 기준으로 동작 |
| **1단계 완료 직후** | 2단계 탭 자동 활성화. 3단계 탭도 즉시 사용 가능 (1단계 원본 기준 체크) |
| **레퍼런스 프리셋 적용** | 2단계 슬라이더 자동 설정 + **3단계 룰셋 자동 선택 + 체크 자동 실행** (원클릭 전체 파이프라인) |

> 체크리스트 자동 재실행은 Tab 전환 시점에 **비동기**로 실행되어, 기획자가 결과 목록을 보는 동안 백그라운드에서 완료된다. 진행 상황은 프로그레스 바로 표시.

## 2단계 — 기획의도에 따라 지형·지물 수정

**프리셋 + 파라미터 슬라이더**로 1단계에서 생성된 실제 맵을 의도 기반 수정.  
무에서 생성하지 않는다. 항상 1단계 결과물(실제 지형) 위에 적용한다.

슬라이더·프리셋 조작 시 내부적으로 **Edit Layer**(비파괴 레이어)가 자동 생성된다. 기획자는 생성된 레이어를 목록에서 보기/숨기기·삭제할 수 있다.

### 2.1 의도 파라미터 — 슬라이더

기존 맵의 건물·도로·지형을 일괄 조정하는 슬라이더.

| 파라미터 | 범위 | 동작 |
|----------|------|------|
| **도심 밀도** | 0~100 | 현재값보다 낮추면 건물 일부 자동 제거, 높이면 빈 필지에 건물 추가 |
| **개방도** | 0~100 | 높이면 시야 차단 건물 선별 제거 + 잔해 대체, 낮추면 빈 공간에 건물 추가 |
| **동선 복잡도** | 0~100 | 높이면 우회 도로 추가, 낮추면 비주요 도로 차단 |
| **고저차 강조** | 0~100 | 높이면 기존 언덕 고도↑ + 계곡 고도↓, 낮추면 전체 평탄화 방향 |
| **파괴도** | 0~100 | 기존 건물의 destruction_state 비율 조정 (0=intact, 100=대부분 destroyed) |

**동작 원리**: 슬라이더 변경 시 기존 맵 데이터를 분석하여 2단계 Edit Layer를 **자동 생성**. 기획자는 생성된 레이어를 개별 검토·수정·삭제 가능.

#### 슬라이더 초기값 산출 — 맵 로드 시 자동 분석

맵을 처음 열거나 1단계 생성 직후, 기존 데이터를 분석하여 5개 슬라이더의 "현재 맵 상태"를 0~100으로 환산한다.

| 슬라이더 | 입력 데이터 | 산출 공식 | 예시 |
|----------|-----------|----------|------|
| **도심 밀도** | `buildings.json` 건물 수, 맵 면적 | `min(100, 건물수 / (맵면적_km² × 50) × 100)` | 3km² 맵에 80건물 → 80/(3×50)×100 = **53** |
| **개방도** | 200m 그리드 4방향 시선 분석 | `(시야 비차단 방향 수 / 전체 방향 수) × 100` | 총 400방향 중 240 개방 → **60** |
| **동선 복잡도** | `roads.json` 노드·엣지 수 | `min(100, 평균_노드_degree / 4 × 100)` | 평균 degree 2.4 → 2.4/4×100 = **60** |
| **고저차** | `heightmap.png` 샘플링 | `min(100, (최고-최저) / (맵반경_m × 0.1) × 100)` | 고도차 30m, 반경 1500m → 30/150×100 = **20** |
| **파괴도** | `buildings.json` destruction_state | `(partial+destroyed 건물 수 / 전체 건물 수) × 100` | 80건물 중 파괴 8 → **10** |

> 산출은 1단계 데이터(JSON + heightmap)만 사용하므로 < 1초. 레퍼런스 프리셋 적용 시에는 프리셋 값이 초기값을 덮어쓴다.

#### 슬라이더별 선별 알고리즘

**도심 밀도 (Urban Density)** — 건물 수 증감

| 방향 | 알고리즘 |
|------|---------|
| **감소** (현재값→낮은값) | 1. 전체 건물을 `buildings.json`에서 로드<br>2. 맵 중심에서 거리순 정렬 (먼 건물 = 우선 제거 후보)<br>3. 동일 거리면 바닥 면적 작은 순서<br>4. 감소 비율 = `(현재값 - 목표값) / 현재값` → 해당 비율만큼 건물 선택<br>5. 선택된 건물마다 `building_remove` Edit Layer 생성 |
| **증가** (현재값→높은값) | 1. `FSpatialHash2D`로 건물 미배치 영역(빈 셀) 탐색<br>2. 빈 셀 중 도로 인접(50m 이내) 셀 우선<br>3. 증가 비율만큼 빈 셀에 `building_add` Edit Layer 생성<br>4. 형상은 아래 **건물 추가 형상 결정 규칙** 참조 |

**건물 추가 형상 결정 규칙** — `building_add` 레이어의 메쉬·크기·높이 결정

2단계에서 추가하는 건물은 OSM 데이터가 없으므로, 인접 건물 분석으로 형상을 결정한다.

| 속성 | 결정 방법 | 폴백 |
|------|----------|------|
| **TypeKey** | 반경 200m 내 인접 건물의 TypeKey **최빈값** | `Residential_Mid` |
| **Footprint** | 인접 건물 바닥면적의 **중앙값** (극단값 배제) | 10m × 10m |
| **높이** | 인접 건물 높이의 **중앙값** | 8m (2층) |
| **회전** | 인접 도로 방향에 **평행 정렬** (도로 Normal ±5° 랜덤) | 0° |
| **파괴 상태** | 주변 50m 건물 파괴 비율에 비례하여 확률 결정 | `intact` |

메쉬 매핑 — `TypeKey → FBuildingMeshEntry` (기존 `ULevelToolBuildingPool` 재사용)

| TypeKey | 대표 메쉬 | 높이 범위 | 비고 |
|---------|----------|----------|------|
| `Residential_Low` | 저층 주거 박스 | 3~6m | 단독주택 |
| `Residential_Mid` | 중층 아파트 박스 | 8~20m | 범용 주거 |
| `Commercial` | 상가 박스 (넓은 footprint) | 4~12m | 상업 |
| `Industrial` | 공장/창고 (넓고 낮음) | 6~15m | 산업 |
| `Public` | 대형 건물 | 10~25m | 공공시설 |

> 1단계 OSM 데이터의 `TypeKey` 분포가 맵마다 다르므로, 추가 건물도 해당 맵의 분포를 자연스럽게 따른다.

**건물 배치 충돌 해소** — `building_add` 시 기존 건물·도로와 겹침 방지

| 단계 | 동작 | 상세 |
|------|------|------|
| 1 | 후보 셀 선택 | `FSpatialHash2D`로 빈 셀 탐색 (도로 인접 50m 우선) |
| 2 | AABB 겹침 검사 | 후보 위치에 건물 footprint AABB 배치 → 기존 건물 AABB와 **최소 이격 5m** 검사 |
| 3-a | 겹침 없음 | 그대로 배치 |
| 3-b | 겹침 발생 | 후보 위치를 **4방향(N/E/S/W) × 10m 씩 이동**하여 재검사 (최대 3회, 반경 30m) |
| 3-c | 도로 겹침 | 후보 footprint가 도로 폴리곤과 교차 → **해당 방향으로 도로 폭 + 3m 이격** |
| 4 | 실패 | 3회 시도 후에도 배치 불가 → **스킵** + `OnLog(Warning, "Building_add 배치 실패: {위치}")` |

| 파라미터 | 값 | 근거 |
|---------|-----|------|
| 최소 건물 간 이격 | 5m | OSM 데이터 평균 건물 간격의 최솟값 |
| 최소 도로 이격 | 도로 폭 + 3m | 인도·보도 여유 공간 |
| 재시도 이동 간격 | 10m | footprint 중앙값(10m)의 1배 |
| 최대 재시도 | 3회 (4방향 × 3 = 12 후보) | 성능 vs 배치 성공률 균형 |

> 배치 실패는 대시보드 프리뷰에 반영: "밀도 53→70: 건물 15개 추가 예정, **2개 배치 실패 예상**". 슬라이더 목표값 대비 실제 달성률이 90% 미만이면 Warning 표시.

#### 슬라이더 달성률 표시

실사 기반 맵은 OSM 데이터에 의존하므로, 밀도·개방도 등 건물 증감 슬라이더의 목표값을 100% 달성하지 못할 수 있다. 이를 기획자에게 투명하게 보여준다.

**슬라이더 UI 확장**:

```
도심 밀도    [■■■■■■■□□□] 42     목표: 45  달성: 93%
             ⚠ 빈 공간 부족 (배치 실패 4건)
```

| 표시 조건 | UI 동작 |
|----------|--------|
| 달성률 ≥ 95% | 목표값만 표시 (정상) |
| 달성률 90~95% | 목표/달성 표시 + 연한 노랑 바 |
| 달성률 < 90% | 목표/달성 표시 + ⚠ Warning + 주황 바 + "빈 공간 부족" 사유 |
| 감소 방향 (제거) | 항상 100% 달성 (제거는 빈 공간 무관) |

**사전 예측** (드래그 중, 경량 시뮬레이션):

밀도 슬라이더를 드래그하는 동안, 경량 모드로 빈 셀 수를 빠르게 카운트하여 **예상 달성률**을 표시한다.

```
도심 밀도    [■■■■■■■■□□] 드래그 중: 55
             예상 배치: ~180/220 (82%)  ← 빈 셀 180개 / 필요 220개
```

> 이를 통해 기획자는 슬라이더를 릴리즈하기 전에 "이 값으로 올리면 얼마나 채워지는지" 미리 파악하고, 다른 슬라이더(개방도 감소 → 빈 셀 증가)로 보완 가능.

**개방도 (Openness)** — 시야 차단 건물 선별 제거

| 단계 | 동작 |
|------|------|
| 1 | 맵을 200m 그리드로 분할, 각 그리드 중심에서 4방향(N/E/S/W) 시선 추적 |
| 2 | 시선 추적 = 그리드 중심에서 해당 방향 200m 직선상에 `buildings.json` 건물 AABB가 교차하는지 판정 |
| 3 | 4방향 중 3방향 이상 차단하는 건물 = "시야 차단 건물"로 태깅 |
| 4 | 개방도 증가 시: 시야 차단 건물 중 높이 낮은 순서로 제거 비율만큼 `building_remove` 생성 |
| 5 | 개방도 감소 시: 시선이 탁 트인 그리드 셀에 `building_add` 생성 |

**동선 복잡도 (Route Complexity)** — 도로 네트워크 조정

| 방향 | 알고리즘 |
|------|---------|
| **증가** | `roads.json` 그래프에서 degree-2 이상 노드(교차로) 사이에 우회 경로가 없는 구간 탐색 → 해당 구간 옆에 `road_add` Edit Layer 생성 |
| **감소** | `roads.json` 그래프에서 degree-1 노드(막다른 길) 또는 전체 교통량에 기여가 낮은 말단 도로 → `road_block` Edit Layer 생성 |

**고저차 강조 (Elevation Contrast)** — 지형 고도 조정

| 방향 | 알고리즘 |
|------|---------|
| **증가** | 1. Landscape를 200m 그리드 샘플링하여 평균 고도 산출<br>2. 평균 이상 지점 → `terrain_modify(offset, +delta)` / 평균 이하 지점 → `terrain_modify(offset, -delta)`<br>3. delta = `(목표값 - 현재값) / 100 × max_delta` |
| **감소** | 전체 지형에 대해 평균 고도로 `terrain_modify(flatten)` Edit Layer 생성 |

**max_delta 동적 산출**: 맵 규모별 고정값이 아닌 **맵 실제 고도 범위에 비례**하여 결정.

```
max_delta = (맵_최고고도 - 맵_최저고도) × 0.05
```

| 맵 | 고도 범위 | max_delta | 효과 |
|----|----------|-----------|------|
| 서울 종로 (15~342m) | 327m | 16.35m/셀 | 도시 구릉 적절 변형 |
| 알프스 댐 (420~780m) | 360m | 18.0m/셀 | 산악 지형 의미 있는 변형 |
| 태국 푸켓 (0~50m) | 50m | 2.5m/셀 | 저지대 미세 조정 |

> 고도 범위 데이터는 `heightmap.png` 로드 시 min/max 샘플링으로 `map_meta.json`에 `"elevation_range_m": [420, 780]` 필드로 저장한다.

**파괴도 (Destruction Level)** — 건물 상태 일괄 변경

| 방향 | 알고리즘 |
|------|---------|
| **증가** | 전체 건물을 무작위 셔플 → 목표 비율에 도달할 때까지 `destruction_state`를 intact→partial→destroyed 순서로 승격하는 Edit Layer 생성 |
| **감소** | destroyed→partial→intact 순서로 강등. 단, 이미 intact인 건물은 스킵 |

예시: 도심 밀도 70→40
1. 감소 비율 = (70−40)/70 ≈ 43%
2. 전체 건물을 맵 중심 기준 먼 거리순 정렬
3. 상위 43% 건물에 대해 `building_remove` 레이어 자동 생성 (keep_foundation: true)
4. 기획자가 "이 건물은 살리고 싶다" → 해당 레이어만 삭제

#### 슬라이더 동시 적용 — 충돌 해소 규칙

레퍼런스 프리셋이 5개 슬라이더를 동시에 변경할 때, 슬라이더 간 대상이 겹칠 수 있다. 이를 해소하기 위한 **적용 순서와 중복 처리** 규칙:

**적용 순서** (의존성 기반, 고정)

```
① 고저차 → ② 파괴도 → ③ 도심 밀도 → ④ 개방도 → ⑤ 동선 복잡도
```

| 순서 | 슬라이더 | 이유 |
|------|---------|------|
| ① | 고저차 | 지형 변경이 최하위 → 건물/도로 기반을 먼저 확정 |
| ② | 파괴도 | 파괴 상태 변경은 건물 존재 여부에 영향 없음 → 밀도·개방도 전에 처리 |
| ③ | 도심 밀도 | 건물 수 증감이 개방도 산출에 영향 → 밀도 먼저 |
| ④ | 개방도 | ③에서 확정된 건물 목록 기준으로 시야 차단 분석 |
| ⑤ | 동선 복잡도 | 건물 배치 완료 후 도로 네트워크 조정 |

**중복 대상 처리**

| 상황 | 처리 |
|------|------|
| ③ 밀도에서 제거한 건물을 ④ 개방도도 제거 후보로 선택 | **스킵** — 이미 제거된 건물은 ④ 후보에서 자동 제외 |
| ③ 밀도에서 추가한 건물이 ④ 개방도에서 시야 차단 | **제거 가능** — 새로 추가한 건물도 개방도 분석 대상에 포함 |
| ④ 개방도 변경 후 슬라이더 초기값이 달라짐 | **재산출 안 함** — 동시 적용은 원본 상태 기준으로 전체 계산. 적용 완료 후 초기값 1회 갱신 |

구현: 각 슬라이더 알고리즘은 **유효 맵 상태**(공통 사항 참조)를 입력으로 받는다. ①→⑤ 순서로 적용하면서, 각 단계 완료 후 유효 상태를 재구성하고 다음 단계에 전달한다.

```
effective_state = RebuildEffectiveState(base + existing_layers)
for slider in [고저차, 파괴도, 도심밀도, 개방도, 동선복잡도]:
    new_layers = slider.Generate(effective_state, target_value)
    add_layers(new_layers)
    effective_state = RebuildEffectiveState(base + all_layers)
```

### 2.2 프리셋 — 편집 레이어 자동 생성 세트

#### 프리셋 인터랙션 모델

프리셋은 **대상 선택 → 프리셋 버튼 클릭**의 2단계로 동작한다.

| 선택 타입 | 뷰포트 조작 | 용도 |
|-----------|------------|------|
| **건물 선택** | 뷰포트에서 건물 Actor 클릭 → 하이라이트 | 저격 포인트 강조, 수직 전투 강조 |
| **영역 선택** | 뷰포트에서 드래그로 원형 영역 지정 (중심+반경) | 중앙 광장 개방 |
| **도로 선택** | 뷰포트에서 도로 Actor 클릭 → 하이라이트 | 진입로 좁히기, 병목+우회로 |
| **없음** | 선택 불필요 (맵 전체 대상) | 레퍼런스 프리셋, 슬라이더 |

> 이 선택 도구는 "편집 도구"가 아니라 **프리셋 적용 대상을 지정하는 선택기**이다. 선택 후 프리셋 버튼을 누르면 Edit Layer가 자동 생성되며, 기획자가 대상을 직접 수정하는 것이 아니다.

#### 뷰포트 프리뷰 — 적용 전 시각 피드백

프리셋·슬라이더·레퍼런스 프리셋 적용 전에, 기획자가 뷰포트에서 변경 결과를 미리 확인할 수 있다.

| 트리거 | 뷰포트 표시 | 비용 |
|--------|-----------|------|
| **프리셋 버튼 호버** (0.5초 이상) | 제거 대상 건물 = **빨간 반투명**, 추가 위치 = **초록 반투명 와이어프레임** | < 200ms (대상 산출만, Actor 미생성) |
| **슬라이더 드래그 중** | 현재 드래그 값 기준 제거/추가 대상 하이라이트 (위와 동일) | < 200ms |
| **레퍼런스 프리셋 호버** | 5개 슬라이더 변경 전체의 제거/추가 대상 하이라이트 | < 500ms |
| **마우스 떠남 / 드래그 취소** | 하이라이트 즉시 해제 | — |
| **클릭 / 릴리즈 (적용)** | 하이라이트 해제 → 실제 레이어 생성 | 기존 성능 예산 |

구현: 프리뷰는 실제 Actor를 생성/삭제하지 않고, `ULineBatchComponent` 또는 기존 Actor의 `SetRenderCustomDepth(true)` + 커스텀 스텐실 값으로 하이라이트. 호버 해제 시 즉시 복원.

> 프리뷰는 **대시보드 수치 프리뷰**와 동시에 작동한다. 기획자는 뷰포트(시각)와 대시보드(수치) 양쪽에서 변경 영향을 확인한 뒤 적용 여부를 결정할 수 있다.

#### 경량 시뮬레이션 모드 — 프리뷰 전용

레퍼런스 프리셋 호버 시 5개 슬라이더를 전부 시뮬레이션하면 정규 알고리즘 5회 = 최대 1~2초가 소요된다. 프리뷰는 **정확도보다 속도**가 중요하므로, 경량 모드를 사용한다.

| 항목 | 정규 모드 (적용 시) | 경량 모드 (프리뷰 시) |
|------|-------------------|---------------------|
| 건물 선별 | 전체 건물 정렬+필터 | **샘플링**: 전체의 20% 무작위 추출 → 대상 수 역산 |
| 시야 분석 | 200m 그리드 4방향 전체 | **4배 거친 그리드** (400m/800m) → 셀 수 1/4 |
| 도로 그래프 | 전체 BFS | **스킵**: 제거/추가 대상 도로 수만 카운트 |
| 지형 시뮬레이션 | Heightmap 실제 수정 | **스킵**: 영향 영역 바운드만 표시 (등고선 오버레이) |
| 충돌 검사 | Spatial Hash 충돌 판정 | **스킵**: 빈 셀 탐색만 (겹침 허용) |

성능 목표:

| 맵 크기 | 건물 수 | 경량 시뮬레이션 | 정규 시뮬레이션 |
|---------|--------|-------------|--------------|
| 1km (소형) | ~80 | < 50ms | < 1초 |
| 3km (중형) | ~200 | < 150ms | < 3초 |
| 5km (대형) | ~500 | < 300ms | < 5초 |

경량 모드 결과는 "~N개 건물 제거, ~M개 추가 예상"으로 **근사치**를 표시한다. 뷰포트 하이라이트 대상도 샘플링 결과 기준이므로 실제 적용 시와 위치가 다를 수 있다. UI에 `"프리뷰 (근사)"` 라벨을 표시.

#### 슬라이더 값 도출 근거

각 프리셋의 슬라이더 수치는 레퍼런스 맵의 실제 특성에서 역산했다.

| 슬라이더 | 측정 기준 | 역산 방법 |
|----------|----------|----------|
| **도심 밀도** | 맵 전체 면적 대비 건물 점유율 | Erangel 8×8km 중 건물 밀집 지역 ≈ 15~20% → 40. Sanhok 4×4km 전체 밀집 → 70 |
| **개방도** | 주요 이동 경로에서 100m 이상 시선이 트이는 비율 | Miramar 사막 구간 80%+ 개방 → 80. Buried City 골목 밀집 → 20 |
| **동선 복잡도** | 두 거점 간 유효 경로 수 (1=직선, 3+=복잡) | Sanhok 밀집 교차로 평균 3.2경로 → 70. Spaceport 직선 위주 → 40 |
| **고저차** | 맵 내 최고-최저 고도차 / 맵 반경 비율 | Dam 수직 80m+ → 80. Spaceport 평지 → 30. Miramar 능선 → 70 |
| **파괴도** | 전체 건물 중 파괴/손상 상태 건물 비율 | Buried City 50%+ 파괴 → 50. Erangel 대부분 intact → 10 |

> 이 값은 "정확한 측정치"가 아니라 **레퍼런스 맵 플레이 경험과 공개 맵 데이터에서 도출한 근사치**로, 기획자가 프리셋 적용 후 세부 조정하는 것을 전제로 한다.

#### 추천 좌표 예시

기획자가 직접 좌표를 입력할 수도 있지만, 프리셋 선택 시 테마에 맞는 실제 좌표를 제안한다.

### 2.4 커스텀 프리셋 저장

기획자가 만든 레이어 그룹 또는 레퍼런스 프리셋 수정본을 "프리셋으로 저장" → JSON 내보내기.

#### 저장 포맷

```json
{
  "preset_name": "종로_시가전_v2",
  "author": "designer_kim",
  "created_at": "2026-03-19T14:30:00Z",
  "base_reference": "Erangel 스타일",
  "sliders": {
    "urban_density": 55,
    "openness": 45,
    "route_complexity": 60,
    "elevation_emphasis": 40,
    "destruction": 20
  },
  "ruleset": "battle_royale",
  "object_presets_applied": ["병목+우회로 세트", "수직 전투 강조"],
  "description": "종로 기반 시가전 강화 버전. Erangel에서 밀도+15, 개방도-15 조정"
}
```

저장 경로: `{ProjectDir}/Saved/LevelTool/CustomPresets/{preset_name}.json`

#### 저장 시 포함/제외 항목

| 포함 | 제외 | 이유 |
|------|------|------|
| 슬라이더 5값 | 개별 Edit Layer 상세 | 레이어는 맵 종속이므로 다른 맵에 재적용 불가 |
| 적용한 오브젝트 프리셋 목록 | 제안 카드 수락 내역 | 제안은 맵 구조 종속 |
| 룰셋 선택 (BR/EX) | 좌표 | 좌표는 맵마다 다름 |
| 기반 레퍼런스 프리셋명 | 체크 결과 | 체크는 실시간 산출 |
| 작성자·날짜·설명 | — | 팀 공유 시 식별용 |

#### 로드 시 검증

| 검증 항목 | 실패 시 처리 |
|----------|-----------|
| JSON 스키마 유효성 | 파싱 실패 → Warning 토스트 + 로드 중단 |
| 슬라이더 값 범위 (0~100) | 범위 초과 → `clamp(0, 100)` + Warning |
| 오브젝트 프리셋명 존재 확인 | 미존재 프리셋 → 스킵 + "'{이름}' 프리셋을 찾을 수 없습니다" |
| `ruleset` 값 | "battle_royale" 또는 "extraction" 외 → 기본 BR |

#### 팀 공유

| 방식 | 절차 |
|------|------|
| **파일 공유** | `CustomPresets/` 폴더를 프로젝트 VCS(Git/Perforce)에 포함 → 팀원이 자동으로 프리셋 목록에서 확인 |
| **내보내기** | [내보내기] 버튼 → 지정 경로에 `.json` 복사 (이메일·메신저 공유용) |
| **가져오기** | [가져오기] 버튼 → `.json` 선택 → `CustomPresets/`에 복사 + 프리셋 목록 갱신 |

> 커스텀 프리셋은 **슬라이더 값 + 메타정보**만 저장하므로 파일 크기 < 1KB. 팀 내 수십 개를 공유해도 부담 없다.

맵 구조를 분석하여 선택한 룰셋에 어울리는지 **체크리스트로 진단**하고, 문제 지점에 대한 **편집 옵션을 제안**한다. AI가 자동으로 고치는 것이 아니라, 기획자에게 선택지를 보여준다.

| EX-9 | 소음 마커 3개+ (다양한 유형) | 소음 마커 1~2개 | 마커 0개 (⬜ 미설정) |
| EX-10 | AI 존 2개+ (다른 위협 수준) | AI 존 1개 | 마커 0개 (⬜ 미설정) |

### 3.4 체크 진단 동작 방식

```
[기획자: 룰셋 선택 (BR 또는 Extraction)]
  ↓
[시스템: 맵 데이터 자동 분석]
  - 거점 자동 탐지 (건물 밀집 클러스터 → S/A/B/C 등급 추정)
  - 건물 분포 밀도 계산
  - 도로 네트워크 연결성 확인
  - 지형 고저차 분석
  ↓
[체크리스트 결과 표시]
  - Pass ✅ / Fail ❌ / Warning ⚠️ / N/A (데이터 없음)
  - Fail 항목에 제안 카드 표시
  ↓
[기획자: 제안 카드 수락/거절]
  - 수락 → 해당 Edit Layer 자동 생성
  - 거절 → 무시
  - 이후 2단계 슬라이더/프리셋으로 추가 조정
```

#### EX 체크리스트 2단계 진단 전략

EX 체크리스트 10항 중 5항(EX-1, EX-5, EX-8, EX-9, EX-10)은 **마커 존재를 전제**로 한다. 맵에 마커가 없는 상태(첫 진단)에서는 이 항목이 항상 Fail/N/A로 나오므로, 진단을 **2단계로 나눠서** 기획자 혼란을 방지한다.

```
[1차 진단: 구조 체크]  ← 마커 없이 분석 가능한 항목
  EX-1~4, EX-6, EX-7   (지형·건물·도로 기반 6항)
  ↓
  Fail 항목 제안 카드 표시 → 기획자 수락/거절
  ↓
[시스템: 마커 미배치 감지]
  EX-5, EX-8, EX-9, EX-10 → "마커 필요" 안내 + 자동 제안
  "추출 포인트가 설정되지 않았습니다. 배치를 제안합니다."
  ↓
[기획자: 마커 제안 수락/거절]
  ↓
[2차 진단: 마커 체크]  ← 마커 배치 후 실행
  EX-1(추출 경로 재분석), EX-5, EX-8, EX-9, EX-10
```

UI 표시:

| 마커 미배치 시 | 표시 | 기획자 행동 |
|-------------|------|-----------|
| 추출 마커 없음 | `EX-1: N/A (추출 미설정)`, `EX-5: N/A` | 제안 카드로 추출 배치 |
| 잠금 마커 없음 | `EX-8: ⬜ 미설정 (선택 사항)` | 제안 수락 또는 의도적 스킵 |
| 소음/AI 없음 | `EX-9: ⬜ 미설정`, `EX-10: ⬜ 미설정` | 제안 수락 또는 의도적 스킵 |

> BR 체크리스트는 마커 의존 항목이 BR-10(탐험 리워드) 1건뿐이므로 별도 단계 없이 "⬜ 미설정" 표시로 충분하다.

**거점 자동 탐지**: `buildings.json`에서 건물 중심 좌표를 추출하고, DBSCAN 클러스터링으로 거점을 그룹핑한다.

| 파라미터 | 값 | 근거 |
|----------|-----|------|
| `eps` (탐색 반경) | 150m | Erangel 도시 거점 내 건물 간격 평균 100~200m |
| `min_samples` (최소 건물 수) | 5 | 5개 미만은 독립 건물이지 거점이 아님 |

클러스터링 후, 클러스터 내 **건물 수 × 평균 바닥면적**으로 "거점 스코어"를 산출하고 등급을 부여한다.

| 등급 | 거점 스코어 조건 | 레퍼런스 대응 |
|------|----------------|-------------|
| **S** | 상위 10% | Erangel Military Base, Buried City Hospital |
| **A** | 상위 10~30% | Erangel Yasnaya Polyana, Dam Power Plant |
| **B** | 상위 30~60% | Erangel Rozhok, Spaceport Container Storage |
| **C** | 하위 40% | Erangel 농가, Blue Gate Standard Passage |

기획자는 탐지된 거점 목록과 등급을 확인만 한다. 등급이 부정확하면 슬라이더(도심 밀도)로 건물 분포를 조정하여 간접적으로 등급을 변경할 수 있다.

#### 수역 횡단 경로 인식

체크리스트의 경로 분석(BR-3, BR-7, EX-6)에서 **수역(강, 호수)은 조건부 횡단 가능 경로**로 취급한다. `water.json`의 water body 데이터를 활용한다.

| 수역 조건 | 횡단 가능 여부 | 경로 가중치 | 근거 |
|----------|--------------|-----------|------|
| 폭 ≤ 30m (하천·개울) | ✅ 도보 횡단 | 도로의 3배 | PUBG/Tarkov: 얕은 하천 수영 가능 |
| 폭 30~100m (중형 강) | ⚠️ 수영 가능 (느림) | 도로의 6배 | 지연·노출 위험 존재 |
| 폭 > 100m (대형 호수·바다) | ❌ 차단 (교량 필요) | ∞ | 보트 없이 불가 |

경로 그래프 구성 시:
1. `water.json`에서 각 water body의 폴리곤을 로드
2. 폴리곤을 가로지르는 두 지점 간 최단 횡단 폭을 계산
3. 횡단 폭 기준으로 경로 가중치를 부여하여 그래프 엣지에 추가
4. 도로 엣지(가중치 1.0)와 수역 엣지(가중치 3.0/6.0)를 통합한 그래프로 경로 분석

> 수역 횡단을 "경로 있음"으로 인정하더라도 높은 가중치로 인해 체크리스트는 **도로/다리 경로를 우선 권장**한다. 수역만으로 연결된 경로는 Warning으로 표시.

### 3.5 제안 카드 형식

```json
{
  "check_id": "BR-3",
  "result": "fail",
  "problem": "서쪽 다리를 차단하면 Military Base 영역이 고립됩니다",
  "suggested_fix": {
    "type": "road_add",
    "area": { "type": "path", "points_ue5": [[...], [...]] },
    "params": { "road_type": "minor", "width_m": 4 },
    "label": "[제안] Military Base 우회 도로"
  },
  "reference": "Erangel: 동·서 다리 + 하부 캣워크로 병목 우회 보장"
}
```

- 각 제안 카드에 **레퍼런스 근거** 표시
- [수락] → Edit Layer로 추가 (created_by: "ai_suggest")
- 기획자가 나중에 토글/삭제/수정 가능

#### 제안 카드 좌표 생성 알고리즘

제안 카드의 `area.points_ue5` 좌표는 체크리스트 진단 결과에서 **문제 지점을 자동 특정**하여 산출한다.

| 체크 유형 | 좌표 생성 방법 | 예시 |
|----------|-------------|------|
| **도로 추가** (BR-3, EX-1, EX-6) | 도로 그래프에서 단절 간선(bridge edge) 양쪽 노드를 시작·끝점으로 설정 → 두 노드 사이 **지형 경사가 가장 낮은 경로**를 Landscape 높이맵 기반 A* 탐색 (3~5개 웨이포인트) | 다리 단절 → 강 하류 300m 지점 낮은 지형으로 우회 경로 |
| **건물 제거** (BR-4, EX-5) | 유효 맵 상태에서 밀도 과다 셀의 중심 좌표 → 해당 셀 내 **바닥면적 최소** 건물의 `actor_ref` | 밀집 셀의 가장 작은 건물 제거 |
| **건물 추가** (EX-4, EX-7) | 노출 구간(200m+ 연속 건물 없음)의 **중간 지점** ±50m에 `point` 좌표 생성 | 300m 노출 → 150m 지점에 엄폐 건물 |
| **능선/지형** (BR-5, EX-3) | 거점 주변 **가장 평탄한 200m 구간**의 중심을 시작점, 해당 구간 양 끝을 path 포인트로 설정 | 평탄지에 능선 추가 |
| **마커 부여** (BR-10, EX-8~10) | 대상 거점 클러스터의 **무게 중심 좌표** (건물 면적 가중 평균) | S등급 거점 중심에 잠금 마커 |

도로 추가 경로 상세 — A* 기반 웨이포인트 생성:

```
입력: 단절 노드 A, B (UE5 좌표)
1. A-B 직선을 샘플링 간격으로 샘플링 (맵 규모별 아래 참조)
2. 각 샘플 지점에서 Landscape 높이 + 건물 AABB 충돌 검사
3. 충돌 지점을 회피하도록 좌우 50m 이내 대체 지점 탐색
4. 경사도 = |높이 차 / 수평 거리|, 경사도 최소 경로 선택
5. 경유점 3~5개로 단순화 (Douglas-Peucker ε=10m)
출력: path.points_ue5 = [[Ax,Ay], [W1x,W1y], ..., [Bx,By]]
```

**대형 맵 성능 스케일링**: 5km 맵에서 10m 간격 시 A-B 거리 1km = 100 샘플 × 건물 충돌 검사. Spatial Hash 사용 시 O(1) 충돌이므로 문제없으나, 제안 카드 여러 개가 동시 생성될 때 병렬화가 필요하다.

| 맵 반경 | 샘플 간격 | 최대 A-B 거리 | 최대 샘플 수 | 단일 경로 비용 | 성능 예산 |
|---------|----------|-------------|------------|-------------|---------|
| 1km | 10m | ~500m | 50 | < 5ms | < 50ms (10경로) |
| 3km | 20m | ~1.5km | 75 | < 8ms | < 80ms (10경로) |
| 5km | 30m | ~3km | 100 | < 12ms | < 120ms (10경로) |

> 샘플 간격을 맵 반경에 비례하여 조정(`interval = max(10, 맵반경_m / 100)`)하면 샘플 수를 ~100으로 제한. Douglas-Peucker의 ε도 비례 조정(`ε = interval`). 전체 제안 카드(최대 10건)의 도로 경로 생성은 **< 200ms** 목표.

> 모든 좌표는 **유효 맵 상태** 기준으로 산출. 이미 제거된 건물 위치는 빈 공간으로 처리, 이미 추가된 건물은 충돌 대상에 포함.

### 3.6 진단 결과 시각화 — 히트맵

체크 결과를 맵 탑뷰에 색상 오버레이로 표시.

| 표시 모드 | 색상 | 용도 |
|-----------|------|------|
| 건물 밀도 | 빨강(과밀)~초록(적정) | BR-4, EX-4 |
| 도로 연결성 | 빨강(단절)~초록(2+경로) | BR-3, BR-5, EX-1, EX-6 |
| 거점 분포 | 마커 위치 + 반경 표시 | BR-1, BR-2, EX-2 |
| 고저차 | 등고선 오버레이 | EX-3 |

**구현**: 분석 결과를 `UTexture2D::CreateTransient`로 RGBA 텍스처 생성 → `ADecalActor`로 투영. (기존 LevelTool heightmap 프리뷰 패턴 재사용)

### 3.7 진단 데이터 수집 — 간이 분석 (v1.1 대비 축소)

v1.1에서 셀당 108회 Raycast를 계획했으나, **과도**하다고 판단. 간이 방식으로 축소.

| 분석 항목 | 방법 | 비용 |
|-----------|------|------|
| 건물 밀도 | 맵을 50m 격자로 분할 → 각 셀에 겹치는 건물 수 카운트 | buildings.json 파싱만, Raycast 없음 |
| 도로 연결성 | roads.json의 도로 그래프에서 BFS/DFS | JSON 그래프 탐색, O(V+E) |
| 거점 거리 | 거점 마커 간 유클리드 거리 | 단순 계산 |
| 고저차 | heightmap에서 거점 위치 높이 샘플링 | 이미지 픽셀 읽기 |
| 가시선 (간이) | 거점 간 직선 위에 건물 존재 여부 확인 | 건물 footprint 교차 검사 |

**전체 분석 시간 목표**: 모든 맵 크기에서 **< 3초**. Raycast 없이 JSON 데이터 분석만으로 수행.

### 3.8 맵 적합도 대시보드

슬라이더·프리셋·제안 카드 조작 시, 기획자가 **"지금 어떤 상태인지"**와 **"바꾸면 어떻게 되는지"**를 한눈에 파악할 수 있는 상시 표시 패널.

#### 대시보드 구성 — 2종 지표

| 지표 | 질문 | 산출 방식 |
|------|------|----------|
| **① 설계 기준 적합도** | "이 맵이 BR/EX로서 제대로 동작하는가?" | 체크리스트 항목별 **가중 합산** (레퍼런스 빈도 기반) |
| **② 레퍼런스 유사도** | "이 맵이 어떤 게임 느낌인가?" | 슬라이더 5값을 레퍼런스 프리셋 값과 **유클리드 거리 비교** |

> 이 두 지표는 역할이 다르다. ①은 "문제점 발견 → 수정 가이드", ②는 "방향성 확인 → 의도 이탈 감지". 단순 점수 매기기가 아니라 **기획 의사결정 지원 대시보드**이다.

#### ① 설계 기준 적합도 — 가중 합산

체크리스트 각 항목에 **"몇 개의 레퍼런스 맵에서 공통으로 나타나는가"**를 가중치로 부여.

**BR 체크리스트 가중치 (6맵 기준)**

| 항목 | 공통 맵 수 | 가중치 | 근거 |
|------|-----------|--------|------|
| BR-1 거점 분산 | 6/6 | 15 | Erangel~Verdansk 전부 S~C 등급 이격 설계 |
| BR-2 루트 그래디언트 | 6/6 | 15 | 전부 중앙=위험=고루트 |
| BR-3 병목 우회로 | 6/6 | 15 | 전부 병목+대안 경로 쌍 |
| BR-4 개방/폐쇄 혼합 | 6/6 | 12 | 전부 교전 거리 다양화 |
| BR-5 엄폐물 테마 | 6/6 | 8 | 전부 테마별 엄폐물 존재 |
| BR-6 차량 이동 경로 | 5/6 | 8 | Sanhok 제외 (도보 가능) |

| | | **합계 100** | |

**채점**: Pass=가중치 × 1.0, Warning=가중치 × 0.5, Fail=0

> **적합도 100점의 의미**: 적합도 100은 "완벽한 맵"이 아니라, **"체크리스트 전 항목이 Pass"**임을 뜻한다. 체크리스트는 레퍼런스 맵에서 공통으로 관찰되는 **최소 설계 기준**을 검증하므로, 100점 = "기본기가 갖춰진 맵". 기획자의 독창적 판단(예: 의도적 병목, 극단적 밀도)은 체크리스트 점수 밖의 영역이며, 대시보드는 이를 방해하지 않고 **왜 감점인지 사유만 제시**한다. 점수가 낮아도 의도된 설계라면 무시해도 좋다.

**EX 체크리스트 가중치 (9맵 기준)**

| 항목 | 공통 맵 수 | 가중치 | 근거 |
|------|-----------|--------|------|
| EX-1 탈출 루트 복수성 | 9/9 | 15 | 전부 추출 경로 2개+ |
| EX-2 위험=보상 비례 | 9/9 | 15 | 전부 고루트=고위험 |
| EX-3 수직 전투 | 8/9 | 10 | Hunt 제외 (평면적) |

#### ② 레퍼런스 유사도 — 프리셋 거리

현재 맵의 슬라이더 5값과 각 레퍼런스 프리셋 값 사이의 **정규화 유클리드 거리**를 산출.

```
distance = √( Σ ((현재값ᵢ - 프리셋값ᵢ) / 100)² )    (i = 5개 슬라이더)
similarity = max(0, (1 - distance / √5)) × 100%
```

| 예시 | 현재 맵 (밀도,개방,동선,고저차,파괴) | Erangel (40,60,50,40,10) | 거리 | 유사도 |
|------|-------------------------------------|--------------------------|------|--------|
| 프리셋 직후 | (40,60,50,40,10) | — | 0 | **100%** |
| 밀도 +15 | (55,60,50,40,10) | — | 0.15 | **93%** |
| 밀도 +30, 개방 -20 | (70,40,50,40,10) | — | 0.36 | **84%** |

표시: **"가장 유사: Erangel (84%)"** + 2순위 **"Sanhok (71%)"**

#### 대시보드 ↔ 뷰포트 인과 연결

대시보드에서 체크 항목을 클릭하면, 해당 항목의 **원인 지점을 뷰포트에서 시각적으로 강조**한다. 기획자가 "왜 점수가 떨어졌는지"를 공간적으로 즉시 파악할 수 있다.

| 대시보드 클릭 대상 | 뷰포트 동작 | 예시 |
|------------------|-----------|------|
| **감점 항목** (예: ❌ BR-3) | 카메라를 문제 지점으로 **자동 이동** + 해당 영역 **주황 원형 하이라이트** | BR-3 클릭 → 단절 다리 위치로 카메라 이동, 단절 간선 양쪽 50m 원형 표시 |
| **제안 카드** | 제안 카드의 `area.points_ue5`를 뷰포트에 **초록 점선으로 표시** + 카메라 이동 | 우회 도로 제안 → 도로 경로 점선 + 시작·끝점 마커 |
| **개선 항목** (예: +6 밀도) | 변경으로 영향받는 **건물 목록 하이라이트** (추가=초록, 제거=빨강) | 밀도 +6 클릭 → 추가 예정 건물 15개 초록 하이라이트 |
| **레퍼런스 유사도 바** | 해당 레퍼런스 프리셋의 추천 좌표 위치를 미니맵에 **핀 표시** | Erangel 84% 클릭 → 미니맵에 상트페테르부르크 좌표 핀 |

구현: 체크리스트 결과(`check_report.json`)의 각 항목에 `"focus_location_ue5": [x,y,z]`와 `"highlight_actors": ["Building_042", ...]` 필드를 포함시킨다. 대시보드 클릭 시 `FEditorViewportClient::SetViewLocation()` + `SetRenderCustomDepth` 호출.

> "보기만 하는 대시보드"가 아닌 **"클릭하면 문제 지점으로 안내하는 대시보드"**. 이로써 수치(대시보드)→공간(뷰포트)→행동(슬라이더/프리셋) 흐름이 끊기지 않는다.

#### 변경 전 프리뷰 (시뮬레이션)

슬라이더를 드래그하거나 프리셋 버튼에 마우스를 올릴 때, **적용하지 않고** 예상 결과를 표시.

| 시점 | 동작 | 비용 |
|------|------|------|
| **슬라이더 드래그 중** | 슬라이더 값만으로 ② 유사도 즉시 계산 (5차원 거리) | < 1ms |
| **슬라이더 릴리즈 전** | 변경될 Edit Layer를 JSON 레벨에서 시뮬레이션 → ① 적합도 예상 | < 500ms |
| **적용 후** | 실제 Actor 배치 후 체크리스트 실행 → ① 적합도 확정 | < 3초 |

> **유사도는 슬라이더 값만으로 산출**하므로 실시간 프리뷰 가능. **적합도는 체크리스트 시뮬레이션**이 필요하므로 릴리즈 시 또는 적용 후 갱신.

#### 오차 범위

예상 적합도와 확정 적합도 사이에 오차가 발생할 수 있다.

| 오차 원인 | 영향 | 대응 |
|----------|------|------|
| 건물 배치 충돌 (일부 스킵) | 밀도·개방도 차이 | 예상값에 "±5" 오차 범위 표시 |
| 지형 간섭 (도로 경사) | 동선 복잡도 차이 | 동일 |
| 건물 제거 연쇄 효과 | 시야 변경 | 동일 |

UI 표시: **"적합도: ~78 (±5)"** → 적용 후 **"적합도: 75 (확정)"**

#### 예측 오차 보정

반복 사용 시 예측 정확도를 높이기 위한 자동 보정:

| 메커니즘 | 동작 | 조건 |
|---------|------|------|
| **이력 기반 보정** | 최근 5회 적용의 (예상-확정) 오차 평균을 다음 예측에 보정값으로 반영 | `보정된 예상 = 예상값 + 평균_오차` |
| **충돌률 학습** | 슬라이더 적용 시 건물 배치 충돌 비율을 기록 → 다음 예측에서 "추가 예정 건물 수 × (1 - 충돌률)"로 보정 | 밀도 증가 시에만 |
| **오차 범위 동적 조정** | 이력 오차의 표준편차를 기반으로 ±범위를 동적 조정 (최소 ±2, 최대 ±10) | 적용 5회 이상 후 |

이력 저장: `map_meta.json`의 `"prediction_history": [{"predicted": 78, "actual": 75}, ...]`

> 보정은 **같은 맵 내**에서만 유효. 새 맵에서는 이력 초기화되어 기본 ±5에서 시작.

### 내부 메커니즘 — Edit Layer Stack

기획자가 직접 다루지 않지만, 슬라이더·프리셋·제안 카드 조작 시 내부적으로 생성되는 비파괴 레이어 시스템.

#### 레이어 스키마 (`layers.json`)

```json
{
  "version": "3.0",
  "map_id": "종로_base_20260319",
  "layers": [
    {
      "layer_id": "uuid-...",
      "type": "terrain_modify | building_add | building_remove | building_height | road_block | road_add | road_width | destruction_state | marker",
      "area": {
        "type": "polygon | circle | point | path | actor_ref",
        "data": {}
      },
      "params": {},
      "label": "자동 생성 이름 (예: [슬라이더] 밀도 70→40 건물제거 #3)",
      "locked": false,
      "visible": true,
      "created_by": "slider | preset | reference_preset | ai_suggest"
    }
  ]
}
```

#### `area.data` 하위 스키마

| area.type | data 포맷 | 용도 |
|-----------|----------|------|
| `polygon` | `{ "points_ue5": [[x,y], [x,y], ...] }` | 영역 선택 (광장 개방 등) |
| `circle` | `{ "center_ue5": [x,y], "radius_cm": 10000 }` | 반경 선택 (슬라이더 지형 수정) |
| `point` | `{ "location_ue5": [x,y,z] }` | 건물 추가 위치 |
| `path` | `{ "points_ue5": [[x,y], [x,y], ...], "width_cm": 600 }` | 도로 추가/능선 경로 |
| `actor_ref` | `{ "actor_id": "Building_042" }` | 기존 건물/도로 참조 |

#### type별 params 정의

| type | params | 설명 |
|------|--------|------|
| `terrain_modify` | `{ "operation": "offset\|flatten\|ridge", "height_delta_cm": 500, "falloff": "smooth", "strength": 0.8 }` | 지형 고도 변경 |
| `building_add` | `{ "type_key": "Residential_Mid", "height_m": 15, "rotation_deg": 45, "destruction_state": "intact" }` | 건물 생성 |
| `building_remove` | `{ "keep_foundation": false }` | 건물 제거 |
| `building_height` | `{ "scale_factor": 1.5 }` | 건물 Z 스케일 |
| `destruction_state` | `{ "new_state": "partial\|destroyed" }` | 파괴 상태 변경 |
| `road_block` | `{ "obstacle_type": "barricade\|rubble\|vehicle", "passable_on_foot": true }` | 도로 장애물 |
| `road_add` | `{ "road_type": "major\|minor\|path", "width_m": 6 }` | 도로 생성 |
| `road_width` | `{ "width_delta_m": -2, "min_width_m": 2 }` | 도로 폭 변경 |
| `marker` | `{ "marker_type": "...", ... }` (아래 마커 상세 참조) | 메타데이터 마커 |

| `extraction_point` | `{ "marker_type": "extraction_point", "label": "북쪽 추출" }` | 추출/탈출 지점 | EX-1, EX-5 |
| `lock_marker` | `{ "marker_type": "lock_marker", "lock_type": "key\|fuel\|code", "target_actor": "Building_034" }` | 열쇠/잠금 필요 지점 | EX-8, BR-10 |
| `noise_trigger` | `{ "marker_type": "noise_trigger", "noise_type": "glass\|birds\|metal_floor\|water_splash", "radius_m": 20 }` | 소음 발생 지점 | EX-9 |

#### 지형 스냅샷 복원

Landscape Heightmap 수정은 본질적으로 파괴적이므로, `terrain_modify` 레이어 적용 시 **스냅샷**을 저장하여 비파괴성을 보장한다.

| 시점 | 동작 | 데이터 |
|------|------|--------|
| 적용 **전** | 영향 영역 Heightmap 텍셀 복사 | `snapshot_before`: 영역 좌표 + uint16 높이값 배열 |
| 적용 **후** | Landscape에 반영 | `snapshot_after`: 수정된 높이값 배열 |
| 레이어 **숨김** | `snapshot_before`로 Heightmap 복원 | — |
| 레이어 **보이기** | `snapshot_after`로 재적용 | — |
| 레이어 **삭제** | `snapshot_before`로 복원 후 스냅샷 삭제 | — |

저장 경로: `{ProjectDir}/Saved/LevelTool/EditLayers/{map_id}/snapshots/{layer_id}.bin`  
포맷: 바이너리 (영역 바운드 + uint16 배열). 50m 반경 기준 약 10~50 KB.

> 지형 레이어가 여러 개 겹칠 경우, 토글 시 **전체 지형 레이어를 아래→위 순서로 재적용**한다.

#### 지형-건물/도로 Z 동기화

`terrain_modify` 레이어 적용 시, 영향 영역 내 건물·도로 Actor의 Z 좌표가 지형 변화에 맞지 않으면 공중 부유 또는 매몰이 발생한다.

**동기화 규칙**:

| 대상 | 동기화 방법 | 시점 |
|------|-----------|------|
| **기존 건물** (1단계) | 건물 pivot의 XY 좌표에서 `ALandscapeProxy::GetHeightAtLocation()` → 건물 Z 좌표 갱신 | `terrain_modify` 적용 직후 |
| **추가 건물** (2단계) | 배치 시점에 이미 수정된 지형 기준으로 Z 산출 → 추가 동기화 불필요 | `building_add` 적용 시 |
| **도로** | `ProceduralMeshComponent` 버텍스의 Z를 Landscape 높이에 재스냅 (Spline 기반: 컨트롤 포인트 Z 갱신 → 메쉬 리빌드) | `terrain_modify` 적용 직후 |
| **terrain_modify 숨김/삭제** | 스냅샷 복원 후, 영향 영역 건물·도로 Z를 **원래 높이로 복원** | 토글/삭제 시 |

구현: `terrain_modify` 레이어에 `affected_actors[]` 배열을 런타임으로 관리. 적용 시 영향 영역 내 Actor를 수집하고, 복원 시 원래 Z를 되돌린다.

```
// terrain_modify 적용 후
for actor in GetActorsInArea(layer.area):
    actor.original_z = actor.GetActorLocation().Z
    new_z = Landscape->GetHeightAtLocation(actor.XY)
    actor.SetActorLocation(FVector(actor.X, actor.Y, new_z))
```

#### 파괴 상태 시각 표현

`destruction_state` 변경 시 건물의 시각적 외관이 달라져야 한다.

| 상태 | 메쉬 처리 | 추가 요소 |
|------|----------|----------|
| `intact` | 원본 `FBuildingMeshEntry` 메쉬 그대로 | — |
| `partial` | 원본 메쉬 유지 + **스케일 Z×0.85** (상층부 손상 표현) + 상단에 **잔해 데칼** 적용 | 건물 기저에 `SM_Rubble_Small` 1~2개 스폰 |
| `destroyed` | 원본 메쉬를 **SM_Foundation** (바닥 기초)로 교체 + 잔해 더미 **SM_Rubble_Large** 배치 | `keep_foundation=true`면 기초 유지, `false`면 완전 제거 |

잔해 메쉬 매핑 (`ULevelToolBuildingPool` 확장):

| 메쉬 ID | 용도 | 크기 |
|---------|------|------|
| `SM_Foundation` | 파괴 건물 바닥 기초 | 원본 footprint 동일 |
| `SM_Rubble_Small` | 부분 파괴 잔해 | 2m × 2m × 1m |
| `SM_Rubble_Large` | 완전 파괴 잔해 더미 | 원본 footprint × 0.5, 높이 2~3m |

> 잔해 메쉬는 **엄폐물 역할**을 하므로 게임플레이에 영향. BR/EX 체크리스트의 "엄폐물 테마 적합성" 항목에서 잔해도 엄폐물로 카운트한다.

#### Undo/Redo

모든 레이어 조작은 `FScopedTransaction`으로 감싸서 Ctrl+Z/Y 지원.

**Transaction 그룹핑 단위**: 하나의 기획자 행동 = 하나의 Undo 단위.

| 기획자 행동 | 생성 레이어 수 | Ctrl+Z 1회 결과 |
|-----------|-------------|----------------|
| 슬라이더 드래그→릴리즈 | N개 (건물 제거/추가) | N개 레이어 **전체 취소** |
| 프리셋 버튼 클릭 | 3~5개 (세트) | 프리셋 세트 **전체 취소** |
| 레퍼런스 프리셋 적용 | 5개 (슬라이더 5종) | 5개 레이어 **전체 취소** |
| 제안 카드 [수락] | 1개 | 해당 레이어 취소 |
| 레이어 보기/숨기기 토글 | 0 (상태 변경) | 토글 상태 복원 |

구현: `FScopedTransaction` 1개를 기획자 행동 시작 시 생성하고, 해당 행동에서 생성되는 모든 레이어를 같은 Transaction 안에 포함시킨다.

#### 유효 맵 상태 (Effective Map State) 재구성

2~3단계의 모든 분석(슬라이더 초기값 산출, 체크리스트 진단, 히트맵 생성, 대시보드 산출)은 **1단계 원본 데이터가 아니라 Edit Layer가 반영된 유효 상태**를 기준으로 수행해야 한다.

```
[1단계 원본]          [Edit Layer Stack]          [유효 맵 상태]
buildings.json   +   building_add/remove/height   →  effective_buildings[]
roads.json       +   road_add/block/width         →  effective_roads (graph)
heightmap.png    +   terrain_modify snapshots      →  effective_heightmap
```

**재구성 절차** (`UEditLayerManager::RebuildEffectiveState`)

| 단계 | 동작 | 산출물 |
|------|------|--------|
| 1 | `buildings.json` 로드 → 전체 건물 목록 생성 | `TArray<FEffectiveBuilding>` |
| 2 | `visible=true`인 레이어를 아래→위 순서로 순회 | — |
| 3-a | `building_remove` → 대상 건물을 목록에서 제거 (또는 `keep_foundation` 시 상태만 변경) | 건물 감소 |
| 3-b | `building_add` → 새 건물을 목록에 추가 | 건물 증가 |
| 3-c | `building_height` → 대상 건물의 높이 스케일 갱신 | 높이 변경 |
| 3-d | `destruction_state` → 대상 건물의 파괴 상태 갱신 | 상태 변경 |
| 4 | `roads.json` 로드 → 도로 그래프 생성 | `FEffectiveRoadGraph` |
| 5-a | `road_block` → 해당 간선의 `passable` 속성 변경 (foot-only 또는 완전 차단) | 간선 수정 |
| 5-b | `road_add` → 새 노드+간선 추가 | 그래프 확장 |
| 5-c | `road_width` → 해당 간선의 폭 속성 변경 | 속성 수정 |
| 6 | Landscape Heightmap은 이미 스냅샷으로 실시간 반영 → 추가 재구성 불필요 | — |

**호출 시점**:

| 이벤트 | 재구성 대상 | 사유 |
|--------|-----------|------|
| 슬라이더 초기값 산출 | buildings + roads | 현재 상태 기준 0~100 환산 |
| 체크리스트 진단 실행 | buildings + roads | 유효 건물·도로 기준 분석 |
| 히트맵 생성 | buildings + roads | 유효 데이터 기준 시각화 |
| 대시보드 적합도 확정 | buildings + roads | 유효 데이터 기준 채점 |
| 대시보드 적합도 **시뮬레이션** | buildings + roads (가상) | 레이어 적용 전 가상 재구성 |

**성능**: 건물 500개 + 도로 3000 엣지 기준 재구성 < 100ms. JSON 파싱이 아닌 메모리 내 `TArray` 순회이므로 캐싱 불필요. 레이어 변경 시마다 즉시 재구성.

**캐싱 전략**: 레이어 변경이 없으면 이전 결과를 재사용. `layers.json`의 해시(레이어 수 + 마지막 layer_id)를 캐시 키로 사용.

#### 레이어 누적/병합 정책

슬라이더를 반복 조정하면 "제거 → 추가 → 다시 제거" 레이어가 누적된다. 관리 복잡도를 제한하기 위한 정책:

| 상황 | 정책 | 예시 |
|------|------|------|
| **동일 슬라이더 재조정** | 이전 슬라이더 레이어를 **교체** (삭제 후 새로 생성) | 밀도 82→40 레이어 삭제 → 밀도 82→55 레이어 새로 생성 |
| **다른 슬라이더 조정** | 별도 레이어 **추가** (독립 관계) | 밀도 레이어 + 개방도 레이어 공존 |
| **프리셋 재적용** | 이전 프리셋 레이어 세트를 **교체** | "저격 포인트 #1,#2,#3" 삭제 → 새 세트 생성 |
| **제안 카드 수락** | 항상 **추가** (독립 레이어) | 기존 레이어에 영향 없음 |

교체 시 `label`의 `created_by` + 슬라이더 ID 조합으로 이전 레이어를 식별한다.

```json
{
  "created_by": "slider",
  "source_id": "urban_density",
  "generation": 2
}
```

`source_id`가 같은 레이어가 이미 존재하면 교체. `generation`은 교체 횟수를 추적한다.

> 이 정책으로 슬라이더 5종 × 1레이어 세트 + 프리셋 N개 + 제안 카드 M개 = **최대 5+N+M개**로 레이어 수가 제한된다.

#### 배치 레이어 — 동일 타입 레이어 일괄화

밀도 슬라이더 1회 조정으로 `building_add` 200개+ 레이어가 생성될 수 있다. 개별 레이어는 건물별 보기/숨기기가 가능하지만, `layers.json` 크기와 직렬화 성능에 부담이 된다. 이를 해소하기 위한 **배치 레이어(Batch Layer)** 옵션:

```json
{
  "layer_id": "uuid-batch-...",
  "type": "building_add_batch",
  "batch_items": [
    { "location_ue5": [x1,y1,z1], "type_key": "Residential_Mid", "height_m": 12, "rotation_deg": 30 },
    { "location_ue5": [x2,y2,z2], "type_key": "Industrial", "height_m": 8, "rotation_deg": 90 },
    ...
  ],
  "label": "[슬라이더] 밀도 10→40 건물추가 (261건)",
  "created_by": "slider",
  "source_id": "urban_density"
}
```

| 모드 | 레이어 수 | 개별 제어 | JSON 크기 | 용도 |
|------|----------|----------|----------|------|
| **개별 모드** (기본) | 건물당 1개 | ✅ 건물별 보기/숨기기·삭제 | 큼 (261 레이어) | 세밀한 조정이 필요할 때 |
| **배치 모드** | 슬라이더당 1개 | ❌ 전체 일괄 ON/OFF만 | 작음 (1 레이어) | 빠른 프로토타이핑, 대형 맵 |

전환: 레이어 그룹 헤더의 `[일괄화]` 버튼 → 그룹 내 동일 타입 레이어를 1개 배치 레이어로 병합. 역방향 `[개별화]` 버튼으로 복원 가능.

> 배치 레이어는 직렬화 시 JSON 크기를 ~1/10로 줄인다 (261 오브젝트 × 200byte → 1 오브젝트 × ~52KB). 대형 맵(5km)에서는 배치 모드를 기본으로 권장.

### 맵 규모별 파라미터 스케일링

LevelTool이 생성하는 맵 반경(1~5km)에 따라, 슬라이더 알고리즘·체크리스트 기준·DBSCAN 파라미터를 자동 조정한다.

| 파라미터 | 반경 1km (소형/EX) | 반경 2~3km (중형) | 반경 5km (대형/BR) | 스케일링 근거 |
|----------|-------------------|-------------------|-------------------|-------------|
| 슬라이더 그리드 셀 크기 | 100m | 200m | 400m | 셀 수 고정(~225셀) → 분석 시간 일정 |
| 도심 밀도 기준 밀도 (건물/km²) | 80 | 50 | 30 | Hunt(1km²)=고밀도, Erangel(64km²)=저밀도 |
| 시야 분석 사거리 | 100m | 200m | 400m | 셀 크기와 동일 |
| DBSCAN `eps` | 80m | 150m | 300m | 건물 간격 비례 (소형맵=밀집) |
| DBSCAN `min_samples` | 3 | 5 | 8 | 맵 크기에 비례한 최소 클러스터 크기 |
| BR-1 거점 최소 이격 | 맵크기×0.3 (300m) | 맵크기×0.2 (500m) | 맵크기×0.2 (1km) | Erangel 기준 2km/8km = 0.25 |
| EX-4 노출 한계 | 100m | 200m | 300m | 맵 크기 비례 |
| 고저차 max_delta | (고도범위×0.05) | (고도범위×0.05) | (고도범위×0.05) | 맵 실제 고도 범위 비례 (2.1 슬라이더 참조) |

스케일링 공식: `value = base_value × (맵반경_km / 2.0)`  
예외: DBSCAN `min_samples`는 `max(3, floor(맵반경_km × 2.5))`

**맵 반경 감지 소스**: 맵 반경은 다음 우선순위로 결정한다.

| 우선순위 | 소스 | 시점 |
|---------|------|------|
| 1 | 1단계 생성 시 기획자가 입력한 `radius_km` 파라미터 | `ULevelToolSubsystem::GenerateMap()` 호출 시 저장 |
| 2 | 레퍼런스 프리셋의 `recommended_coords.radius_km` | 프리셋 적용 시 |
| 3 | Landscape Actor의 실제 크기에서 역산 (`LandscapeSizeX × ScaleX / 100 / 2`) | 폴백: 위 두 값이 없을 때 |

저장: `{ProjectDir}/Saved/LevelTool/EditLayers/{map_id}/map_meta.json`에 `"radius_km": 3.0` 필드로 기록. 2단계 진입 시 자동 로드.

### 레이어 목록 그룹핑 규칙

레이어가 20개 이상 누적될 수 있으므로, UI에서 자동 그룹핑 + 필터링을 제공한다.

**자동 그룹 기준**: 같은 `FScopedTransaction` 내에서 생성된 레이어는 하나의 그룹으로 묶는다.

| 그룹 이름 규칙 | 예시 |
|-------------|------|
| 레퍼런스 프리셋: `"{프리셋명} (N)"` | "Erangel 프리셋 (5)" |
| 편집 프리셋: `"{프리셋명} #{번호} (N)"` | "병목+우회로 #1 (3)" |
| 슬라이더: `"슬라이더: {이름} (N)"` | "슬라이더: 밀도 82→55 (1)" |
| 제안 카드: `"제안 카드 (N)"` | "제안 카드 (5)" |

**그룹 조작**:

| 조작 | 동작 |
|------|------|
| ▶/▼ 클릭 | 접기/펼치기 (기본: 접힘) |
| 그룹 `[숨기기]` | 그룹 내 전체 레이어 `visible=false` |
| 그룹 `[삭제]` | 그룹 내 전체 레이어 삭제 (확인 다이얼로그) |

**필터 버튼**: `created_by` 기준으로 필터. 복수 선택 가능.

| 필터 | 표시 대상 |
|------|----------|
| `[전체]` | 모든 레이어 |
| `[슬라이더]` | `created_by: "slider"` |
| `[프리셋]` | `created_by: "preset" \| "reference_preset"` |
| `[제안]` | `created_by: "ai_suggest"` |
| `[마커]` | `type: "marker"` |

**상단 요약**: `"총 24개 (프리셋 17, 제안 5, 마커 8)"` 형태로 카운트 표시.

### 레퍼런스 확장 절차

신규 게임/맵 레퍼런스를 추가할 때의 갱신 체크리스트.

| 단계 | 작업 | 산출물 |
|------|------|--------|
| 1. 데이터 수집 | 맵 크기, 플레이어 수, POI 목록, 추출/수축 시스템, 고유 메카닉 | 기본 스펙 테이블 |
| 2. 거점 분석 | S~C 등급 분류, 거점별 루트 등급·위험도·전투 특성 | POI 구조 + 상세 거점 테이블 |
| 3. 설계 원칙 도출 | 해당 맵 고유 원칙 5~8항 | 설계 특징 테이블 |
| 4. 비교 테이블 갱신 | BR 또는 EX 비교 요약 테이블에 열 추가 | 비교 요약 업데이트 |
| 5. 공통 원칙 검증 | 기존 공통 원칙(BR 10항/EX 12항)에 신규 근거 추가 또는 신규 원칙 제안 | 공통 원칙 갱신 |
| 6. 프리셋 추가 | 슬라이더 5값 + 추천 좌표 + 룰셋 | 프리셋 JSON + 테이블 행 |
| 7. 체크리스트 검토 | 신규 원칙이 기존 체크항목으로 커버되는지 확인. 미커버 시 항목 추가 | 체크리스트 갱신 |
| 8. 번호 부여 | BR: B{N+1}, EX: E{N+1} 순차 | 섹션 헤더 |
| 9. 문서 버전 업 | 변경 사유·변경 내역·목차 갱신 | 버전 업데이트 |

> 레퍼런스 추가는 **분석 데이터가 기획(프리셋·체크리스트)에 연결되어야** 의미가 있다. 분석만 하고 기획에 반영하지 않는 "고아 레퍼런스"를 방지한다.

### 에러 핸들링

| 상황 | 처리 |
|------|------|
| `layers.json` 파싱 실패 | `OnLog(Error)` 출력 + 빈 Layer Stack으로 초기화. 손상 파일을 `.bak` 으로 백업 |
| 지형 스냅샷 파일 누락 (`.bin`) | 해당 terrain 레이어를 **비활성(visible=false)** 처리 + Warning 토스트. 재생성 불가이므로 레이어 삭제 유도 |
| 슬라이더 적용 중 건물 배치 실패 (충돌/영역 초과) | 해당 건물 스킵 + `OnLog(Warning)` + 제안 카드에 "배치 실패 N건" 표시 |
| 체크리스트 진단 중 `buildings.json` 미존재 | 건물 관련 항목을 "N/A (데이터 없음)"으로 표시. 나머지 항목은 정상 진단 |
| Edit Layer 적용 시 대상 Actor 미존재 | 레이어를 Orphan 상태로 표시 (UI에 ⚠ 아이콘) + 적용 스킵 |
| 프리셋 JSON 스키마 불일치 | `FJsonObjectConverter` 실패 시 기본값 폴백 + Warning 로그 |
