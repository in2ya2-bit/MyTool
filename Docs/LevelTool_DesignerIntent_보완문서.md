## 기획의도 — 3단계 구조

| 단계 | 내용 | 상태 |
|------|------|------|
| **1단계** | 실제 지형을 토대로 한 지형·지물 생성 | ✅ 구현 완료 (기존 LevelTool) |
| **2단계** | 기획의도에 따라 지형·지물 일괄 수정 (슬라이더 + 프리셋) | 신규 개발 |
| **3단계** | 배틀로얄·익스트랙션 룰에 어울리는 옵션 (체크 + 제안) | 신규 개발 |

```
[1단계: 실사 생성] → [2단계: 의도 기반 수정] → [3단계: 룰셋 옵션]
     완료             슬라이더/프리셋          체크리스트+제안
```

### 기획자 워크플로우 시나리오

#### 시나리오 A: "Erangel 느낌의 BR 맵을 처음 만드는 기획자" (~10분)

| 시간 | 단계 | 기획자 행동 | 시스템 반응 |
|------|------|-----------|-----------|
| 0:00 | 1단계 | LevelTool에서 좌표 입력 (또는 추천 좌표 수락) | 실사 지형·건물·도로 생성 |
| 2:00 | 2단계 | "Erangel 스타일" 레퍼런스 프리셋 클릭 | 슬라이더 5개 자동 설정 (밀도40, 개방도60...) → Edit Layer 자동 생성 |
| 3:00 | 2단계 | 뷰포트에서 결과 확인. "도심이 너무 비어보인다" → 밀도 슬라이더 40→55 조정 | 건물 추가 레이어 자동 생성 |
| 4:00 | 2단계 | "이 언덕을 더 강조하고 싶다" → 고저차 슬라이더 40→60 | 언덕↑ 계곡↓ 레이어 생성 |
| 5:00 | 2단계 | "중앙에 광장이 필요하다" → 영역 드래그 → "중앙 광장 개방" 프리셋 클릭 | 건물 제거 + 평탄화 + 도로 확장 레이어 생성 |
| 6:00 | 3단계 | "BR 체크" 탭 → Battle Royale 룰셋 선택 | 자동 진단: BR-3 Fail (병목 우회로 없음) |
| 7:00 | 3단계 | 제안 카드 확인: "서쪽 다리 고립 위험 → 우회 도로 추가" → [수락] | 우회 도로 Edit Layer 생성 |
| 8:00 | 3단계 | 히트맵(건물밀도) 켜서 분포 확인 | 탑뷰 오버레이 표시 |
| 9:00 | 2단계 | 레이어 목록에서 "건물 제거 #7" 이 아깝다 → 숨기기 | 해당 건물 복원 |
| 10:00 | — | 만족 → 커스텀 프리셋으로 저장 | JSON 내보내기 |

### R1. PUBG — Erangel (배틀로얄 레퍼런스)

> 출처: [pubgmap.io](https://pubgmap.io/ko/erangel.html), [PUBG 공식](https://www.pubg.com/en/game-info/maps/erangel)

#### 기본 스펙

| 항목 | 값 |
|------|-----|
| 맵 크기 | 8×8 km (64 km²) |
| 육지 비율 | 51.47% |
| 수역 비율 | 48.53% |
| 플레이어 수 | 100명 |
| 루트 스폰 | 24,058개 |
| 차량 스폰 | 633개 |
| 도로 커버리지 | 4.28% |
| 식생 커버리지 | 30.64% |
| 낙하 가능 영역 | 58.71% |

#### POI 구조 (거점 등급)

| 등급 | 거점 | 특징 |
|------|------|------|
| **S (핫드롭)** | Sosnovka Military Base, Pochinki, School | 최고 루트 + 최고 위험. 3~5팀 경쟁 |
| **A (고루트)** | Georgopol (컨테이너), Novorepnoye, Yasnaya Polyana, Mylta Power | 안정적 루팅, 2~3팀 |
| **B (중루트)** | Rozhok, Prison, Mansion + Shelter, Ferry Pier | 중간 위험, 솔로·듀오 적합 |
| **C (저위험)** | Gatka, Stalber, 농가 산포 | 안전 루팅, 루트 품질 낮음 |

#### 동선 및 병목 구조

| 요소 | 설명 |
|------|------|
| **2개 다리 (동·서)** | 본토 ↔ Military Island 유일한 차량 이동 경로. 맵 최대 병목 |
| **다리 하부 캣워크** | 다리 캠핑 대응용 우회 경로 (Update 11.2 추가) |
| **도로 네트워크** | 전체 면적의 4.28%. 차량 이동 필수 (8km 맵 도보 불가) |
| **자기장 수축** | 원형 수축, 임의 중심. 후반으로 갈수록 이동 강제 |
| **지형 다양성** | 초원, 산악, 도시, 농지, 해안 혼합. 교전 거리 분포 다양 |

#### 설계 특징 — 룰셋 옵션에 반영할 기준

| 설계 원칙 | Erangel 적용 사례 |
|-----------|------------------|
| 거점 간 거리 확보 | S등급 거점 간 최소 1.5~2km 이격 |
| 병목 + 우회로 공존 | 다리(병목) + 캣워크(우회) + 수영(고위험 우회) |
| 루트 등급 그래디언트 | 위험=고루트, 안전=저루트. 중앙=위험, 외곽=안전 |
| 개방/폐쇄 혼합 | 도시(폐쇄 근접전) + 농지(개방 저격전) 교차 배치 |
| 차량 필요성 | 8km 스케일에서 차량 없이 자기장 이동 불가 → 도로=동선 |
| 수축 경로 형평성 | 어떤 위치든 이동 가능해야 함 → 병목이 완전 차단하면 안 됨 |

#### 상세 거점 분석 (나무위키 기반)

| 거점 | 유형 | 설계 특징 |
|------|------|-----------|
| **Sosnovka Military Base (밀베)** | 분리된 섬의 대규모 군사기지 | 2개 다리로만 진입 → 최강 병목. ㄷ자 생활관(옥상 파밍), 레이더 철탑(고지대 저격), 활주로 격납고. 풍부한 루트 대신 탈출 리스크 극대화 |
| **Pochinki (포친키)** | 중앙 대도시 | 맵 정중앙. 3층 옥상 지붕 뛰기(닌자 플레이). 밀집된 건물+골목 → 시가전 대표 거점. "헬친키" 별명 |
| **School+Apartments (학교/아파트)** | 대형 단일 건물+아파트 단지 | 교실/강당/수영장 복합. 좁은 공간 다층 CQB. 졸업 후 Rozhok/아파트로 연계 진출 |
| **Georgopol (게오르고폴)** | 만 기준 남/북 분리 대도시 | 강남=꿀집+컨테이너항, 강북=9파트 아파트. 같은 도시 내 병목(만) 존재 |
| **Yasnaya Polyana (야스나야)** | 북동쪽 대도시 | 고층 감옥 건물(전략적 요충지), 꿀집 4채, 다양한 파밍 루트. 존버 유저 다수 |
| **Mylta Power (밀타발전소)** | 남동쪽 끝 발전소 | 좁은 면적 고밀도 루트. 굴뚝=극한 저격 포지션. 주변 폐공장 연계 파밍 |
| **Shelter (대피소)** | 지하 벙커 | 지상 표시 없는 숨겨진 4출입구 대형 벙커. 미로 구조 → 산탄총/투척무기 강제. 탐험 리워드 설계의 원형 |
| **Quarry (채석장)** | 분지 채석장 | 분지=엄폐 없음 → 고지대 선점 필수. 비밀 지하실(업데이트 추가) |

## 2단계 — 기획의도에 따라 지형·지물 수정

**프리셋 + 파라미터 슬라이더**로 1단계에서 생성된 실제 맵을 의도 기반 수정.  
무에서 생성하지 않는다. 항상 1단계 결과물(실제 지형) 위에 적용한다.

슬라이더·프리셋 조작 시 내부적으로 **Edit Layer**(비파괴 레이어)가 자동 생성된다. 기획자는 생성된 레이어를 목록에서 보기/숨기기·삭제할 수 있다.

### 2.1 의도 파라미터 — 슬라이더

기존 맵의 건물·도로·지형을 일괄 조정하는 슬라이더.

| 파라미터 | 범위 | 동작 |
|----------|------|------|
| **도심 밀도** | 0~100 | 현재값보다 낮추면 건물 일부 자동 제거, 높이면 빈 필지에 건물 추가 |
| **개방도** | 0~100 | 높이면 시야 차단 건물 선별 제거 + 잔해 대체, 낮추면 빈 공간에 건물 추가 |
| **동선 복잡도** | 0~100 | 높이면 우회 도로 추가, 낮추면 비주요 도로 차단 |
| **고저차 강조** | 0~100 | 높이면 기존 언덕 고도↑ + 계곡 고도↓, 낮추면 전체 평탄화 방향 |
| **파괴도** | 0~100 | 기존 건물의 destruction_state 비율 조정 (0=intact, 100=대부분 destroyed) |

**동작 원리**: 슬라이더 변경 시 기존 맵 데이터를 분석하여 2단계 Edit Layer를 **자동 생성**. 기획자는 생성된 레이어를 개별 검토·수정·삭제 가능.

#### 슬라이더 초기값 산출 — 맵 로드 시 자동 분석

맵을 처음 열거나 1단계 생성 직후, 기존 데이터를 분석하여 5개 슬라이더의 "현재 맵 상태"를 0~100으로 환산한다.

| 슬라이더 | 입력 데이터 | 산출 공식 | 예시 |
|----------|-----------|----------|------|
| **도심 밀도** | `buildings.json` 건물 수, 맵 면적 | `min(100, 건물수 / (맵면적_km² × 50) × 100)` | 3km² 맵에 80건물 → 80/(3×50)×100 = **53** |
| **개방도** | 200m 그리드 4방향 시선 분석 | `(시야 비차단 방향 수 / 전체 방향 수) × 100` | 총 400방향 중 240 개방 → **60** |
| **동선 복잡도** | `roads.json` 노드·엣지 수 | `min(100, 평균_노드_degree / 4 × 100)` | 평균 degree 2.4 → 2.4/4×100 = **60** |
| **고저차** | `heightmap.png` 샘플링 | `min(100, (최고-최저) / (맵반경_m × 0.1) × 100)` | 고도차 30m, 반경 1500m → 30/150×100 = **20** |
| **파괴도** | `buildings.json` destruction_state | `(partial+destroyed 건물 수 / 전체 건물 수) × 100` | 80건물 중 파괴 8 → **10** |

> 산출은 1단계 데이터(JSON + heightmap)만 사용하므로 < 1초. 레퍼런스 프리셋 적용 시에는 프리셋 값이 초기값을 덮어쓴다.

#### 슬라이더별 선별 알고리즘

**도심 밀도 (Urban Density)** — 건물 수 증감

| 방향 | 알고리즘 |
|------|---------|
| **감소** (현재값→낮은값) | 1. 전체 건물을 `buildings.json`에서 로드<br>2. 맵 중심에서 거리순 정렬 (먼 건물 = 우선 제거 후보)<br>3. 동일 거리면 바닥 면적 작은 순서<br>4. 감소 비율 = `(현재값 - 목표값) / 현재값` → 해당 비율만큼 건물 선택<br>5. 선택된 건물마다 `building_remove` Edit Layer 생성 |
| **증가** (현재값→높은값) | 1. `FSpatialHash2D`로 건물 미배치 영역(빈 셀) 탐색<br>2. 빈 셀 중 도로 인접(50m 이내) 셀 우선<br>3. 증가 비율만큼 빈 셀에 `building_add` Edit Layer 생성<br>4. TypeKey는 인접 건물의 TypeKey 중 최빈값 사용 |

**개방도 (Openness)** — 시야 차단 건물 선별 제거

| 단계 | 동작 |
|------|------|
| 1 | 맵을 200m 그리드로 분할, 각 그리드 중심에서 4방향(N/E/S/W) 시선 추적 |
| 2 | 시선 추적 = 그리드 중심에서 해당 방향 200m 직선상에 `buildings.json` 건물 AABB가 교차하는지 판정 |
| 3 | 4방향 중 3방향 이상 차단하는 건물 = "시야 차단 건물"로 태깅 |
| 4 | 개방도 증가 시: 시야 차단 건물 중 높이 낮은 순서로 제거 비율만큼 `building_remove` 생성 |
| 5 | 개방도 감소 시: 시선이 탁 트인 그리드 셀에 `building_add` 생성 |

**동선 복잡도 (Route Complexity)** — 도로 네트워크 조정

| 방향 | 알고리즘 |
|------|---------|
| **증가** | `roads.json` 그래프에서 degree-2 이상 노드(교차로) 사이에 우회 경로가 없는 구간 탐색 → 해당 구간 옆에 `road_add` Edit Layer 생성 |
| **감소** | `roads.json` 그래프에서 degree-1 노드(막다른 길) 또는 전체 교통량에 기여가 낮은 말단 도로 → `road_block` Edit Layer 생성 |

**고저차 강조 (Elevation Contrast)** — 지형 고도 조정

| 방향 | 알고리즘 |
|------|---------|
| **증가** | 1. Landscape를 200m 그리드 샘플링하여 평균 고도 산출<br>2. 평균 이상 지점 → `terrain_modify(offset, +delta)` / 평균 이하 지점 → `terrain_modify(offset, -delta)`<br>3. delta = `(목표값 - 현재값) / 100 × max_delta(5m)` |
| **감소** | 전체 지형에 대해 평균 고도로 `terrain_modify(flatten)` Edit Layer 생성 |

**파괴도 (Destruction Level)** — 건물 상태 일괄 변경

| 방향 | 알고리즘 |
|------|---------|
| **증가** | 전체 건물을 무작위 셔플 → 목표 비율에 도달할 때까지 `destruction_state`를 intact→partial→destroyed 순서로 승격하는 Edit Layer 생성 |
| **감소** | destroyed→partial→intact 순서로 강등. 단, 이미 intact인 건물은 스킵 |

예시: 도심 밀도 70→40
1. 감소 비율 = (70−40)/70 ≈ 43%
2. 전체 건물을 맵 중심 기준 먼 거리순 정렬
3. 상위 43% 건물에 대해 `building_remove` 레이어 자동 생성 (keep_foundation: true)
4. 기획자가 "이 건물은 살리고 싶다" → 해당 레이어만 삭제

### 2.2 프리셋 — 편집 레이어 자동 생성 세트

#### 프리셋 인터랙션 모델

프리셋은 **대상 선택 → 프리셋 버튼 클릭**의 2단계로 동작한다.

| 선택 타입 | 뷰포트 조작 | 용도 |
|-----------|------------|------|
| **건물 선택** | 뷰포트에서 건물 Actor 클릭 → 하이라이트 | 저격 포인트 강조, 수직 전투 강조 |
| **영역 선택** | 뷰포트에서 드래그로 원형 영역 지정 (중심+반경) | 중앙 광장 개방 |
| **도로 선택** | 뷰포트에서 도로 Actor 클릭 → 하이라이트 | 진입로 좁히기, 병목+우회로 |
| **없음** | 선택 불필요 (맵 전체 대상) | 레퍼런스 프리셋, 슬라이더 |

> 이 선택 도구는 "편집 도구"가 아니라 **프리셋 적용 대상을 지정하는 선택기**이다. 선택 후 프리셋 버튼을 누르면 Edit Layer가 자동 생성되며, 기획자가 대상을 직접 수정하는 것이 아니다.

#### "저격 포인트 강조" (건물 선택)

| 순서 | 생성 레이어 | 상세 |
|------|------------|------|
| 1 | terrain_modify | 선택 건물 반경 50m 지형 +5m 올림 |
| 2 | building_height | 해당 건물 scale_factor 1.3 |
| 3 | building_remove | 정면 60° 시야각 내 저층 건물 1~2개 제거 |

#### "중앙 광장 개방" (영역 선택 — 드래그)

| 순서 | 생성 레이어 | 상세 |
|------|------------|------|
| 1 | building_remove | 반경 100m 내 건물 일괄 제거 (keep_foundation: true) |
| 2 | terrain_modify | 해당 영역 flatten (영역 평균 높이) |
| 3 | road_width | 광장 연결 도로 2개 폭 +3m |

#### "진입로 좁히기" (도로 선택 — 클릭)

| 순서 | 생성 레이어 | 상세 |
|------|------------|------|
| 1 | road_width | 선택 도로 폭 -3m (최소 2m) |
| 2 | road_block | 도로 양쪽에 vehicle 장애물 |
| 3 | building_add | 도로 측면에 Wall_Segment 2개 추가 |

#### "병목 + 우회로 세트" (도로 선택 — 클릭) — Erangel 다리 설계 참조

| 순서 | 생성 레이어 | 상세 |
|------|------------|------|
| 1 | road_block | 선택 도로에 rubble 장애물 (차량 차단, 도보 통과 가능) |
| 2 | road_add | 장애물 옆으로 좁은 path (폭 3m) 우회로 생성 |
| 3 | building_add | 우회로 입구에 엄폐용 파괴 건물 1개 |

> Erangel 다리 리디자인(Update 11.2)에서 캣워크 추가로 병목 캠핑 대응 경로를 만든 설계 원칙 적용.

#### "수직 전투 강조" — ARC Raiders Dam 참조

| 순서 | 생성 레이어 | 상세 |
|------|------------|------|
| 1 | building_height | 선택 건물 scale_factor 1.5 (시야 확보 고층화) |
| 2 | building_add | 인접 건물 사이에 저층 건물 추가 (2층 연결 매복 포지션) |
| 3 | terrain_modify | 건물 주변 지형 -2m (지상↔건물 고저차 강조) |

### 2.3 레퍼런스 맵 프리셋 — 실제 게임 스타일 원클릭 적용

레퍼런스 게임의 맵 스타일을 재현하는 통합 프리셋. 선택 시 **추천 좌표 + 슬라이더 자동 설정 + 룰셋 자동 선택**이 한번에 적용된다.

#### 동작 흐름

```
[레퍼런스 프리셋 선택: "Erangel 스타일"]
  ↓
1단계: 추천 좌표로 실제 지형 생성 (또는 기획자가 직접 좌표 입력)

2단계: 슬라이더 5개 자동 설정 → Edit Layer 자동 생성
  ↓
3단계: 해당 룰셋 체크리스트 자동 실행
  ↓
기획자: 결과 확인 후 슬라이더/프리셋으로 세부 조정
```

#### 슬라이더 값 도출 근거

각 프리셋의 슬라이더 수치는 레퍼런스 맵의 실제 특성에서 역산했다.

| 슬라이더 | 측정 기준 | 역산 방법 |
|----------|----------|----------|
| **도심 밀도** | 맵 전체 면적 대비 건물 점유율 | Erangel 8×8km 중 건물 밀집 지역 ≈ 15~20% → 40. Sanhok 4×4km 전체 밀집 → 70 |
| **개방도** | 주요 이동 경로에서 100m 이상 시선이 트이는 비율 | Miramar 사막 구간 80%+ 개방 → 80. Buried City 골목 밀집 → 20 |
| **동선 복잡도** | 두 거점 간 유효 경로 수 (1=직선, 3+=복잡) | Sanhok 밀집 교차로 평균 3.2경로 → 70. Spaceport 직선 위주 → 40 |
| **고저차** | 맵 내 최고-최저 고도차 / 맵 반경 비율 | Dam 수직 80m+ → 80. Spaceport 평지 → 30. Miramar 능선 → 70 |
| **파괴도** | 전체 건물 중 파괴/손상 상태 건물 비율 | Buried City 50%+ 파괴 → 50. Erangel 대부분 intact → 10 |

> 이 값은 "정확한 측정치"가 아니라 **레퍼런스 맵 플레이 경험과 공개 맵 데이터에서 도출한 근사치**로, 기획자가 프리셋 적용 후 세부 조정하는 것을 전제로 한다.

#### 추천 좌표 예시

기획자가 직접 좌표를 입력할 수도 있지만, 프리셋 선택 시 테마에 맞는 실제 좌표를 제안한다.

| 프리셋 | 추천 좌표 (위도, 경도) | 반경 | 설명 |

## 3단계 — 배틀로얄·익스트랙션 룰에 어울리는 옵션

맵 구조를 분석하여 선택한 룰셋에 어울리는지 **체크리스트로 진단**하고, 문제 지점에 대한 **편집 옵션을 제안**한다. AI가 자동으로 고치는 것이 아니라, 기획자에게 선택지를 보여준다.

### 3.1 룰셋 선택

| 룰셋 | 레퍼런스 | 핵심 질문 |
|------|---------|-----------|
| **Battle Royale** | PUBG Erangel | "100명이 떨어져서 싸울 때, 이 맵이 공정하고 재미있는가?" |
| **Extraction** | ARC Raiders Dam | "소규모 팀이 루트하고 탈출할 때, 긴장감과 선택지가 충분한가?" |

### 3.2 Battle Royale 체크리스트 — PUBG 4맵 기준

Erangel·Miramar·Sanhok·Vikendi 공통 설계 원칙에서 도출한 체크리스트.

| # | 체크 항목 | 기준 (레퍼런스) | 진단 방법 | 미충족 시 제안 옵션 |
|---|----------|---------------|-----------|-------------------|
| BR-1 | **거점 분산** | S등급 거점 간 최소 맵 크기×0.2 이격 (Erangel: Pochinki↔Military 2km+) | 거점 마커 간 거리 측정 | "거점 {A}와 {B}가 너무 가깝습니다" → [건물 제거로 한쪽 등급 낮추기] |
| BR-2 | **루트 등급 그래디언트** | 중심/위험=고루트, 외곽/안전=저루트 (전 맵 공통) | 거점 마커 위치 vs 맵 중심 거리 | "외곽에 S등급 거점이 있습니다" → [거점 마커 재배치] 또는 [건물 밀도 조정] |
| BR-3 | **병목 우회로** | 모든 병목에 우회 경로 1개 이상 (Erangel: 다리+캣워크, Sanhok: 섬간 다리+수영) | 도로 그래프에서 제거 시 단절되는 간선 식별 | "다리 {X}를 차단하면 고립됩니다" → [우회 도로 추가] 또는 [수영 가능 수역 표시] |
| BR-4 | **개방/폐쇄 혼합** | 건물 밀집 30~50% (Erangel 30%, Miramar 20%, Sanhok 50%) | 건물 분포 밀도 맵 | "전체가 도심입니다 (80%)" → [외곽 건물 제거] |
| BR-5 | **엄폐물 테마 적합성** | 테마별 주력 엄폐물 존재 확인 (사막→능선, 정글→식생, 설원→바위) | 지형 + 건물 분포 교차 확인 | "사막 테마인데 평탄지만 있습니다" → [능선 추가] 또는 [바위 배치] |
| BR-6 | **차량 이동 경로** | 맵 대각선 > 2km이면 도로 연결성 필수 (Miramar: 도로 6.92%) | 거점 간 도로 경로 존재 여부 | "거점 {A}→{B} 도로 없음" → [도로 추가] |
| BR-7 | **수축 형평성** | 맵 어디서든 중심 이동 가능 (완전 차단 병목 금지, 전 맵 공통) | 4방향 → 중심 경로 확인 | "{방향}에서 접근 불가" → [도로 추가] 또는 [건물 제거] |
| BR-8 | **맵 크기별 밀도** | 4km→고밀도, 8km→구역별 밀도 차등 (Sanhok vs Erangel) | 맵 크기 대비 건물 수 비율 | "8km 맵인데 밀도가 Sanhok 수준입니다" → [건물 간격 확대] |

| EX-1 | **탈출 루트 복수성** | 모든 고루트 거점에서 추출까지 경로 2개 이상 (전 맵 공통) | 거점 마커 → 추출 마커 경로 탐색 | "거점 {A}에서 추출까지 경로 1개뿐" → [건물 제거로 우회로 확보] |

### 3.4 체크 진단 동작 방식

```
[기획자: 룰셋 선택 (BR 또는 Extraction)]
  ↓
[시스템: 맵 데이터 자동 분석]
  - 거점 자동 탐지 (건물 밀집 클러스터 → S/A/B/C 등급 추정)
  - 건물 분포 밀도 계산
  - 도로 네트워크 연결성 확인
  - 지형 고저차 분석
  ↓
[체크리스트 결과 표시]
  - Pass ✅ / Fail ❌ / Warning ⚠️
  - Fail 항목에 제안 카드 표시
  ↓
[기획자: 제안 카드 수락/거절]
  - 수락 → 해당 Edit Layer 자동 생성
  - 거절 → 무시
  - 이후 2단계 슬라이더/프리셋으로 추가 조정
```

**거점 자동 탐지**: `buildings.json`에서 건물 중심 좌표를 추출하고, DBSCAN 클러스터링으로 거점을 그룹핑한다.

| 파라미터 | 값 | 근거 |
|----------|-----|------|
| `eps` (탐색 반경) | 150m | Erangel 도시 거점 내 건물 간격 평균 100~200m |
| `min_samples` (최소 건물 수) | 5 | 5개 미만은 독립 건물이지 거점이 아님 |

클러스터링 후, 클러스터 내 **건물 수 × 평균 바닥면적**으로 "거점 스코어"를 산출하고 등급을 부여한다.

| 등급 | 거점 스코어 조건 | 레퍼런스 대응 |
|------|----------------|-------------|
| **S** | 상위 10% | Erangel Military Base, Buried City Hospital |
| **A** | 상위 10~30% | Erangel Yasnaya Polyana, Dam Power Plant |
| **B** | 상위 30~60% | Erangel Rozhok, Spaceport Container Storage |
| **C** | 하위 40% | Erangel 농가, Blue Gate Standard Passage |

기획자는 탐지된 거점 목록과 등급을 확인만 한다. 등급이 부정확하면 슬라이더(도심 밀도)로 건물 분포를 조정하여 간접적으로 등급을 변경할 수 있다.

### 3.5 제안 카드 형식

```json
{
  "check_id": "BR-3",
  "result": "fail",
  "problem": "서쪽 다리를 차단하면 Military Base 영역이 고립됩니다",
  "suggested_fix": {
    "type": "road_add",
    "area": { "type": "path", "points_ue5": [[...], [...]] },
    "params": { "road_type": "minor", "width_m": 4 },
    "label": "[제안] Military Base 우회 도로"
  },
  "reference": "Erangel: 동·서 다리 + 하부 캣워크로 병목 우회 보장"
}
```

- 각 제안 카드에 **레퍼런스 근거** 표시
- [수락] → Edit Layer로 추가 (created_by: "ai_suggest")
- 기획자가 나중에 토글/삭제/수정 가능

### 3.6 진단 결과 시각화 — 히트맵

체크 결과를 맵 탑뷰에 색상 오버레이로 표시.

| 표시 모드 | 색상 | 용도 |
|-----------|------|------|
| 건물 밀도 | 빨강(과밀)~초록(적정) | BR-4, EX-4 |
| 도로 연결성 | 빨강(단절)~초록(2+경로) | BR-3, BR-5, EX-1, EX-6 |
| 거점 분포 | 마커 위치 + 반경 표시 | BR-1, BR-2, EX-2 |
| 고저차 | 등고선 오버레이 | EX-3 |

**구현**: 분석 결과를 `UTexture2D::CreateTransient`로 RGBA 텍스처 생성 → `ADecalActor`로 투영. (기존 LevelTool heightmap 프리뷰 패턴 재사용)

### 3.7 진단 데이터 수집 — 간이 분석 (v1.1 대비 축소)

v1.1에서 셀당 108회 Raycast를 계획했으나, **과도**하다고 판단. 간이 방식으로 축소.

| 분석 항목 | 방법 | 비용 |
|-----------|------|------|
| 건물 밀도 | 맵을 50m 격자로 분할 → 각 셀에 겹치는 건물 수 카운트 | buildings.json 파싱만, Raycast 없음 |
| 도로 연결성 | roads.json의 도로 그래프에서 BFS/DFS | JSON 그래프 탐색, O(V+E) |
| 거점 거리 | 거점 마커 간 유클리드 거리 | 단순 계산 |
| 고저차 | heightmap에서 거점 위치 높이 샘플링 | 이미지 픽셀 읽기 |
| 가시선 (간이) | 거점 간 직선 위에 건물 존재 여부 확인 | 건물 footprint 교차 검사 |

**전체 분석 시간 목표**: 모든 맵 크기에서 **< 3초**. Raycast 없이 JSON 데이터 분석만으로 수행.

## 공통 사항

### 내부 메커니즘 — Edit Layer Stack

기획자가 직접 다루지 않지만, 슬라이더·프리셋·제안 카드 조작 시 내부적으로 생성되는 비파괴 레이어 시스템.

#### 레이어 스키마 (`layers.json`)

```json
{
  "version": "3.0",
  "map_id": "종로_base_20260319",
  "layers": [
    {
      "layer_id": "uuid-...",
      "type": "terrain_modify | building_add | building_remove | building_height | road_block | road_add | road_width | destruction_state",
      "area": {
        "type": "polygon | circle | point | path | actor_ref",
        "data": {}
      },
      "params": {},
      "label": "자동 생성 이름 (예: [슬라이더] 밀도 70→40 건물제거 #3)",
      "locked": false,
      "visible": true,
      "created_by": "slider | preset | reference_preset | ai_suggest"
    }
  ]
}
```

#### `area.data` 하위 스키마

| area.type | data 포맷 | 용도 |
|-----------|----------|------|
| `polygon` | `{ "points_ue5": [[x,y], [x,y], ...] }` | 영역 선택 (광장 개방 등) |
| `circle` | `{ "center_ue5": [x,y], "radius_cm": 10000 }` | 반경 선택 (슬라이더 지형 수정) |
| `point` | `{ "location_ue5": [x,y,z] }` | 건물 추가 위치 |
| `path` | `{ "points_ue5": [[x,y], [x,y], ...], "width_cm": 600 }` | 도로 추가/능선 경로 |
| `actor_ref` | `{ "actor_id": "Building_042" }` | 기존 건물/도로 참조 |

#### type별 params 정의

| type | params | 설명 |
|------|--------|------|
| `terrain_modify` | `{ "operation": "offset\|flatten\|ridge", "height_delta_cm": 500, "falloff": "smooth", "strength": 0.8 }` | 지형 고도 변경 |
| `building_add` | `{ "type_key": "Residential_Mid", "height_m": 15, "rotation_deg": 45, "destruction_state": "intact" }` | 건물 생성 |
| `building_remove` | `{ "keep_foundation": false }` | 건물 제거 |
| `building_height` | `{ "scale_factor": 1.5 }` | 건물 Z 스케일 |
| `destruction_state` | `{ "new_state": "partial\|destroyed" }` | 파괴 상태 변경 |
| `road_block` | `{ "obstacle_type": "barricade\|rubble\|vehicle", "passable_on_foot": true }` | 도로 장애물 |
| `road_add` | `{ "road_type": "major\|minor\|path", "width_m": 6 }` | 도로 생성 |
| `road_width` | `{ "width_delta_m": -2, "min_width_m": 2 }` | 도로 폭 변경 |

#### 지형 스냅샷 복원

Landscape Heightmap 수정은 본질적으로 파괴적이므로, `terrain_modify` 레이어 적용 시 **스냅샷**을 저장하여 비파괴성을 보장한다.

| 시점 | 동작 | 데이터 |
|------|------|--------|
| 적용 **전** | 영향 영역 Heightmap 텍셀 복사 | `snapshot_before`: 영역 좌표 + uint16 높이값 배열 |
| 적용 **후** | Landscape에 반영 | `snapshot_after`: 수정된 높이값 배열 |
| 레이어 **숨김** | `snapshot_before`로 Heightmap 복원 | — |
| 레이어 **보이기** | `snapshot_after`로 재적용 | — |
| 레이어 **삭제** | `snapshot_before`로 복원 후 스냅샷 삭제 | — |

저장 경로: `{ProjectDir}/Saved/LevelTool/EditLayers/{map_id}/snapshots/{layer_id}.bin`

### Phase 전제조건

| Phase | 내용 | 선행 조건 | 기존 코드 활용 |
|-------|------|-----------|---------------|
| **A** | Edit Layer Stack 내부 엔진 + 2단계 슬라이더·프리셋 | 기존 LevelTool 완료 (✅) | `PlaceBuildingsFromJson`, `SpawnRoadActors`, `ClearGeneratedActors`, `FScopedTransaction`, `FBuildingMeshEntry`, `FSpatialHash2D` |
| **B** | 3단계 룰셋 체크리스트 + 히트맵 | Phase A 완료 | `UTexture2D::CreateTransient` (히트맵), `OnProgress`/`OnLog` (진단 진행률) |

> Phase A 즉시 착수 가능. Phase B는 A 완료 후 진행.

### 에러 핸들링

| 상황 | 처리 |
|------|------|
| `layers.json` 파싱 실패 | `OnLog(Error)` 출력 + 빈 Layer Stack으로 초기화. 손상 파일을 `.bak` 으로 백업 |
| 지형 스냅샷 파일 누락 (`.bin`) | 해당 terrain 레이어를 **비활성(visible=false)** 처리 + Warning 토스트. 재생성 불가이므로 레이어 삭제 유도 |
| 슬라이더 적용 중 건물 배치 실패 (충돌/영역 초과) | 해당 건물 스킵 + `OnLog(Warning)` + 제안 카드에 "배치 실패 N건" 표시 |
| 체크리스트 진단 중 `buildings.json` 미존재 | 건물 관련 항목을 "N/A (데이터 없음)"으로 표시. 나머지 항목은 정상 진단 |
| Edit Layer 적용 시 대상 Actor 미존재 | 레이어를 Orphan 상태로 표시 (UI에 ⚠ 아이콘) + 적용 스킵 |
| 프리셋 JSON 스키마 불일치 | `FJsonObjectConverter` 실패 시 기본값 폴백 + Warning 로그 |
